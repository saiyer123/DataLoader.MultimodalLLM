Stateful DataLoader for Mid-Epoch Training Resumption
When training large machine learning models, it is common for training runs to be interrupted due to system crashes, time limits on shared compute clusters, or manual termination. In most modern training pipelines, model parameters, optimizer state, and learning rate scheduler state are saved in checkpoints, allowing training to resume without losing progress. However, a less obvious but equally important component is the data loading process. PyTorch’s default DataLoader does not save its iteration state, meaning that when training resumes, data loading restarts from the beginning of the epoch. This behavior can lead to duplicated samples, skipped samples, and non-reproducible training behavior, especially when data shuffling is enabled. The goal of this work is to design and implement a stateful DataLoader that can resume training from the exact point where it left off, even if the interruption occurs mid-epoch or mid-batch.

In standard PyTorch training loops, the DataLoader controls the order in which dataset samples are provided to the model. When shuffle=True, the DataLoader internally randomizes the order of samples at the start of each epoch. While this improves training stability and generalization, it creates a problem when training is interrupted. Upon restarting, the DataLoader reshuffles the dataset and begins yielding samples from the start of the new shuffled order. As a result, some samples may be processed twice, while others may be skipped entirely. This issue is particularly problematic in long-running jobs or distributed environments where interruptions are common.

The core challenge is that PyTorch’s DataLoader does not expose or persist its internal iteration state. Simply saving the batch index or step number is insufficient, because the shuffled data order may change across runs. Without knowing both the exact order of samples and how far iteration has progressed through that order, it is impossible to resume data loading correctly. Therefore, a robust solution must explicitly manage and store the minimal state necessary to reconstruct the data iteration process.

The key insight behind the proposed solution is that resuming data loading correctly requires only two pieces of information: the fixed order in which dataset samples are presented, and the current position within that order. At the start of each epoch, the DataLoader constructs a list of dataset indices. If shuffling is enabled, this list is randomly permuted using a deterministic seed. This shuffled list defines the complete and fixed order in which samples will be processed for the entire epoch.

As training proceeds, the DataLoader yields samples sequentially from this list, grouping them into batches. Internally, the DataLoader maintains a single counter representing how many samples have already been yielded. This counter increases by one for every sample sent to the model, regardless of batch boundaries. Because batches are simply contiguous slices of the shuffled index list, this single counter implicitly encodes both the current batch number and the position within the batch.

When training is interrupted, the DataLoader’s state is saved by recording the shuffled index list and the current sample position. On resume, the DataLoader reconstructs the same shuffled order and restores the position counter. Iteration then continues from the exact next sample, as if training had never been interrupted. Importantly, no reshuffling occurs on resume, ensuring that the order of samples remains consistent.

This design is implemented by separating data ordering and iteration state from PyTorch’s internal DataLoader machinery. A custom sampler is responsible for generating and iterating over the shuffled index list, starting from a specified position. The sampler exposes a state_dict() method that serializes the index order and current position, and a corresponding load_state_dict() method that restores them. A wrapper DataLoader class rebuilds the underlying PyTorch DataLoader using this sampler whenever state is loaded.

The implementation deliberately tracks state at the sample level rather than the batch level. This choice avoids complications arising from partial batches, variable batch sizes, or configuration options such as drop_last. Because the position counter represents the total number of samples processed so far, resuming mid-batch is handled naturally without additional bookkeeping. To avoid undefined behavior, the implementation explicitly restricts usage to single-worker data loading (num_workers=0), since multi-worker DataLoader semantics introduce concurrency issues that are outside the scope of this assignment.

Correctness was evaluated using a series of deterministic tests designed to verify exact resumption behavior. First, training was partially executed on a synthetic dataset, interrupted mid-epoch, and then resumed using the saved DataLoader state. The combined sequence of samples processed before and after resumption was verified to include each dataset sample exactly once, with no duplication or omission. Second, a deterministic equivalence test compared uninterrupted training to interrupted-and-resumed training under the same random seed. The two sequences of sample indices were shown to be identical, proving that resumption preserves the exact data order. These tests provide strong evidence that the DataLoader correctly resumes from the precise point of interruption.

A smoke test was also performed using a real multimodal dataset to ensure compatibility with realistic data structures. While PyTorch’s default collation mechanism cannot batch variable-length multimodal samples without a custom collate_fn, this limitation is orthogonal to DataLoader statefulness and does not affect the correctness of iteration resumption.
The current implementation intentionally limits support to single-worker data loading, as extending stateful iteration to multi-worker settings would require careful coordination across worker processes. Additionally, batching of variable-length multimodal data requires a custom collation function, which would normally be handled downstream by tokenizers and image processors in a full training pipeline. Future work could extend this approach to distributed training environments and integrate stateful sampling with preprocessing pipelines.
